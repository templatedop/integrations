## Phase 6A: Lookup APIs (10 endpoints)

### Architecture Decision: Static In-Memory Data

Since the DDL defines all lookup values as PostgreSQL ENUMs (not database tables), the Lookup APIs will return **static in-memory data** derived from the enum definitions. No database queries needed except for products lookup.

**Rationale**:
- All lookup values are defined as PostgreSQL ENUMs in the schema
- No separate lookup/reference tables exist in the DDL
- Static data approach is performant (no DB round-trips)
- Enum values are fixed and change only with schema migrations

## Detailed Implementation Specifications

### Lookup Data Sources (from DDL ENUMs)

**LU-POL-002 Relationships** (relationship_enum):
FATHER, MOTHER, SPOUSE, SON, DAUGHTER, BROTHER, SISTER, GRANDFATHER, GRANDMOTHER, UNCLE, AUNT, NEPHEW, NIECE, FRIEND, OTHER

**LU-POL-003 Salutations** (salutation_enum):
MR, MRS, MS, DR, SHRI, SMT, KUM

**LU-POL-006 Age Proof Types** (age_proof_type_enum):
AADHAAR, BIRTH_CERTIFICATE, SCHOOL_CERTIFICATE, PASSPORT, VOTER_ID, DRIVING_LICENSE, PAN, OTHER_STANDARD, NON_STANDARD_AFFIDAVIT, NON_STANDARD_DECLARATION

**LU-POL-007 Payment Modes** (payment_method_enum + swagger extras):
CASH, CHEQUE, DD, ONLINE, POSB, NACH, ECS, STANDING_INSTRUCTION, UPI, NEFT, RTGS

**LU-POL-008 Document Types** (document_type_enum):
PROPOSAL_FORM, DOB_PROOF, ADDRESS_PROOF, PHOTO_ID, MEDICAL_REPORT, PAYMENT_COPY, HEALTH_DECLARATION, PHOTO, INCOME_PROOF, EMPLOYMENT_PROOF, OTHER

### Validation Business Rules

**VAL-POL-001 Eligibility Check**:
- Calculate age from DOB
- Fetch product by code from ProductRepository
- Check: `product.IsEligibleAge(age)`
- Check: `product.IsEligibleSA(sumAssured)`
- Check: policy_term within product limits
- Check: maturity_age <= product.max_maturity_age
- Return array of check results

**VAL-POL-002 Aadhaar Format** (VR-PI-008):
- Pattern: `^[0-9]{12}$`
- Verhoeff checksum validation (optional enhancement)

**VAL-POL-003 PAN Format** (VR-PI-009):
- Pattern: `^[A-Z]{5}[0-9]{4}[A-Z]{1}$`

**VAL-POL-004 Pincode** (VR-PI-034, VR-PI-035):
- Pattern: `^[0-9]{6}$`
- State matching (future: external service integration)

**VAL-POL-005 IFSC** (VR-PI-030):
- Pattern: `^[A-Z]{4}0[A-Z0-9]{6}$`

**VAL-POL-006 Nominee Shares** (VR-PI-018):
- Sum of shares must equal exactly 100.0
- Each share must be > 0 and <= 100

**VAL-POL-007 Date Chain** (BR-POL-018):
- declaration_date <= receipt_date <= indexing_date <= proposal_date
- All dates must be valid format
- Return field-level errors

**VAL-POL-008 Aggregate SA** (INT-POL-002):
- Fetch existing aggregate SA for customer (from ProposalRepository or external Customer Service)
- Check: existing_sa + proposed_sa <= max_allowed
- Return breakdown of current, proposed, max
