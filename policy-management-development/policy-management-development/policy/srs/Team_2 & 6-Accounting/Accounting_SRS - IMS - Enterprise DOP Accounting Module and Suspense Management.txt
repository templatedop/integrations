1. INTRODUCTION
1.1 Purpose
The purpose of this Software Requirement Specification (SRS) is to define the requirements for Transaction posting in DoP Enterprise accounting module. The system shall account, reconcile and post the financial transactions data i.e receipts and payments, maintain accurate records, automate financial reporting, and provide tools for analysis. This will replace a manual or fragmented accounting process to increase efficiency, accuracy, and compliance.
1.2 Scope
In scope: 1. Accounting, Reconciling and Posting of the figures of Receipts and Payments as
reported by the IMS system in accordance with the figures reported by the Receipt/Payment Bank Accounts. 2. Generating various financial reports including collection reports, investible amount report, claims paid reports, incentive paid report, GST reports, TDS report at various levels of hierarchy of the offices operating IMS. 3. Ensuring compliance and maintaining an audit trail for transactions while accounting and posting of the figures.
Out of scope: 1. Functions beyond core accounting activities and functions not specifically
mentioned in this SRS document.
1.3 Intended audience
This document is intended for project managers, software developers, quality assurance (QA) testers, system administrators, business analysts, and financial stakeholders.

2. OVERALL DESCRIPTION
2.1 User roles and characteristics
1. Maker : Responsible for initiating receipt and payment transactions through IMS interface at collection/payment avenues.
2. Approver : Competent authority responsible for approving and authorizing payments through IMS interface at collection/payment avenues and approving modifications done by system administrator.
3. Accountant : Monitors and manages ledger and perform reconciliations and make manual entries to the ledger if required in specific cases.
4. Auditor : Requires read-only access to transaction records and audit trails to verify financial accuracy and compliance.
5. System Administrator : Responsible for system configuration, maintenance, and security. They can modify user accounts, access permissions, chart of accounts (approved by competent authority), mapping of ledger accounts and other technical aspects of the system.
2.2 Functional requirements
2.2.1 General Ledger (GL) management
1. REQ-GL-1.0: The system shall allow System Administrators to set up and customize the chart of accounts (approved by competent authority) and mapping of ledger accounts as per standard accounting procedures as and when approved by the competent authority for implementation.
2. REQ-GL-1.1: The system shall automatically account and post transaction data including IMS receipts and payment through double-entry bookkeeping to the appropriate ledger accounts (IT 2.0 Account Codes.)
3. REQ-GL-1.2: The system shall allow users to post accounting entries in ledger accounts manually in specific cases, with an optional workflow for review and approval.
4

2.2.2 Banking and reconciliation
1. REQ-BANK-2.0: The system shall initiate automatic payments to customer's/agent's bank accounts for incentives, loan and maturity values in accordance with the approved procedures deployed in IMS and as approved by the competent authority.
2. REQ-BANK-2.1: The system shall manage remittance and drawing bank accounts of respective DDOs and reconcile bank statements with recorded transactions automatically.
3. REQ-BANK-2.2: The system shall provide a workflow to reconcile bank statements with recorded transactions manually in specific cases.
2.2.3 Financial reporting
1. REQ-REP-3.0: The system shall generate standard financial statements as and when required, including: I. Detailed and Consolidated Receipt reports with requisite bifurcations. II. Detailed and Consolidated Payment reports with requisite bifurcations. III. Detailed and Consolidated GST and Tax Reports IV. Detailed and Consolidated Investible Funds Reports
2. REQ-REP-3.1: The system shall allow users to export reports in common formats such as CSV, PDF, and Excel.
3. REQ-REP-3.2: The system shall consist of drill-down functionality from summary reports to underlying transaction details.
2.3 Non-functional requirements
1. Performance (NFR-PERF-1.0): The system shall process financial transactions within 2 seconds, and generate standard financial reports within 10 seconds for standard data volumes.
2. Security (NFR-SEC-2.0): The system shall protect all financial data using encryption both in transit and at rest. It must use role-based access control to restrict data
5

access based on user permissions. User authentication must support strong passwords and two-factor authentication (2FA). 3. Reliability (NFR-REL-3.0): The system must operate with 99.9% uptime, excluding planned maintenance. A robust backup and recovery system shall be in place to prevent data loss in the event of a system failure. 4. Usability (NFR-USABILITY-4.0): The user interface must be intuitive, with a clear and consistent navigation design that requires minimal training for typical accounting tasks. 5. Scalability (NFR-SCALE-5.0): The system architecture shall be scalable to accommodate an increase in transaction volume and the number of users as the transaction volume grows. 2.4 External interface requirements 1. User interfaces: The system shall provide a web-based user interface accessible via standard browsers on desktop and mobile devices. 2. Software interfaces: The system shall provide APIs to integrate with third-party software, such as banking portals, payment gateways, and management systems. 3. Communication interfaces: The system shall be capable of sending automated email notifications for financial activities, such as receipts, payments and requests.

3.3 REQ-GL-1.2

Function 5 : Provision for Manual Accounting Entries in the IMS system Sub Function 5.1 : Manual Accounting entries posting functionalities for unaccounted transactions. Description : The IMS system shall provide a dashboard with provisions to manually push the accounting entries of unaccounted IMS transactions in the DoP Accounting module.

Step 1 : The IMS system shall list all the unaccounted IMS transactions with bifurcations if applicable of the respective DDO in the Accountant's dashboard of that DDO. To account a certain transaction or part transaction (bifurcated figure) manually the accountant shall select that transaction or part transaction.

20

Step 2 : The Accountant shall not be able to insert any figure entry while making the accounting entry. The IMS transaction or part transaction to be accounted once selected shall show a drop down menu with the list of all either Receipt or Payment side account code depending on the nature of the transaction.
Step 3 : After selecting the applicable account code the Accountant shall click on the "Initiate Posting" button to initiate the posting of that selected transaction or part transaction.
21

Step 4 : On initiation of the posting for the selected transaction, the transaction shall show on the DDO dashboard for approval, once approved by the DDO on clicking the "Approve Posting" button the transaction shall be posted in the DoP Accounts Module.

3.5 REQ-BANK-2.1 Function 7 : Automatic Reconciliation of Bank statements with IMS statement. Sub Function 7.1 : Automatic Reconciliation of Remittance and Drawing Bank accounts of respective DDOs with IMS receipts and payment statement. Description : The IMS system shall automatically reconcile the daily statements of the Remittances and Drawing bank accounts of the respective DDOs with the receipts and payment statement of IMS for that respective DDO. Step 1 : The IMS systems fetches the Remittance and Drawing bank statements of the respective DDO and the receipts and payment statement of IMS for that respective DDO. Step 2 : The IMS system reconciles the statements with corresponding transactions on both the statements and segregates the non-reconciled transactions in either of the statements for manual reconciliation. Step 3 : The non-reconciled transactions shall be displayed on the respective DDOs Accountant's dashboard for manual reconciliation.

3.6 REQ-BANK-2.2 Function 8 : Provision for posting Manual Reconciliation Entries in the IMS system Sub Function 8.1 : Manual Reconciliation entries posting functionalities for nonreconciled transactions. Description : The IMS system shall provide a dashboard with provisions to manually push the reconciliation entries of non-reconciled IMS transactions in APT 2.0. Step 1 : The IMS system shall list all the non-reconciled transactions on the Accountant's dashboard with bank statement's transaction at one side and the IMS transaction at the other side. The accountant shall be able to select corresponding transaction from both statements for performing reconciliation and no figure entries shall be permitted.
Step 2 : The accountant can click on the "Initiate Reconciliation" once corresponding transactions are selected from both side statements to initiate the posting of the reconciliation entries.

Step 3 : The thus initiated reconciliation entry shall be displayed on the DDOs dashboard for approval, where the entry can be approved by selecting and clicking on "Approve Reconciliation" once approved the reconciliation entry shall be posted in the system.

4. SYSTEM ARCHITECTURE
The system will be based on a micro service and client-server architecture with a modern web interface. Key components will include: 1. Presentation Layer : A web-based application built with a modern front-end framework. 2. Application Layer : A secure, scalable back-end that handles business logic and API requests. 3. Database Layer : A relational database to store and manage all financial data.
27

5. DATA REQUIREMENTS
The database must store critical financial data, including: 1. User profiles and permissions. 2. A complete chart of accounts. 3. Transaction details (date, amount, description, source etc). 4. Salesforce, Doctors and customer information. 5. Receipt and payment records. 6. Financial reports and historical data. 7. All data fields will have defined validation rules to ensure data integrity.
